# Schema 规则文档（版本：1-2026-04-22）

本文档是该版本的 schema 规范快照，用于发布留档与版本对比。

## 1. 文档目标与范围

- 覆盖 `mv-*` 围栏的核心 schema 规则与关键约束。
- 明确“规范来源代码”（source of truth）。
- 记录破坏性变更（breaking changes）与升级建议。

## 2. Source Of Truth

以下文件是 schema 规则的权威来源：

- 类型定义（字段形状）：`packages/core/src/types.ts`
- 解析与校验（合法性规则/报错）：`packages/core/src/parse/blocks.ts`

当本文档与代码不一致时，以以上两个文件为准。

## 3. 围栏类型总览

支持的围栏类型（`MvFenceKind`）：

- `smw-model-sql`
- `smw-model-kv`
- `smw-model-struct`
- `smw-model-codespace`
- `smw-model-interface`
- `smw-view`
- `smw-map`

## 4. 关键规则摘要

### 4.1 `smw-model-sql`

- 顶层需有 `id`（非空字符串）。
- `tables` 必须是非空数组。
- 每张表：
  - `id` 非空且在同块内唯一。
  - `columns` 非空数组；列名唯一。
  - `rows` 为数组；每行不能出现未声明列。
  - 非 `nullable` 列在每行必须存在。
  - 若声明了主键列（`primaryKey: true`），其主键组合在行内不能重复。

### 4.2 `smw-model-kv`

- 顶层需有 `id`（非空字符串）。
- `documents` 必须是数组，且每项必须是 JSON 对象（不能是数组/null）。

### 4.3 `smw-model-struct`

- 顶层需有 `id`。
- `root` 必须是合法组节点（`name` 非空）。
- `groups` / `datasets` 若存在必须是数组，元素结构必须合法。

### 4.4 `smw-model-interface`

- 顶层需有 `id`。
- `endpoints` 必须是非空数组。
- 每个 endpoint 必须有唯一 `id` 和非空 `name`。

### 4.5 `smw-view`

- 顶层需有 `id`、`kind`、`modelRefs[]`。
- `payload` 可为字符串或 JSON 对象（按 `kind` 解释）。
- `uml-*` 的 payload 会进一步校验 `smw-uml/v1` 的结构一致性。
- 对 `uml-class`：
  - `relations.kind = association` 用于成员/属性级关联（通过 `members[]/properties[]` 的 `associatedClassifierId` 同步）。
  - `relations.kind = dependency` 用于 class 级关系（同步到 `namespaces[].associations[]`）。
  - `association` 可携带来源槽位元信息：`fromSlotSection ('members'|'properties')`、`fromSlotName`。

### 4.6 `smw-map`

- 必须包含 `id` 与 `rules[]`。
- 每条规则至少要有 `modelId` 与 `targetPath`。

## 5. `smw-model-codespace` 详细规范

## 5.1 顶层

- 必填：`id`、`modules`（非空数组）
- 可选：`title`、`workspaceRoot`

## 5.2 `modules[]`

- 每项必填：`id`、`name`
- 可选：`path`、`role`、`notes`、`namespaces`
- `id` 在同一围栏内全局唯一（与 namespace/class/function 等共享唯一域）

## 5.3 `namespaces[]`（递归）

- 每项必填：`id`、`name`
- 可选：`qualifiedName`、`notes`
- 可递归包含：
  - `namespaces[]`
  - `classes[]`
  - `variables[]`
  - `functions[]`
  - `macros[]`
  - `associations[]`

## 5.4 `classes[]`

- 必填：`id`、`name`
- 可选：`kind`（`class|interface|struct`）、`abstract`、`stereotype`、`templateParams`、`notes`
- 可选关系与成员：
  - `bases[]`
  - `members[]`
  - `methods[]`
  - `enums[]`
  - `properties[]`

### 5.4.1 `bases[]`

- 每项必填：`targetId`、`relation`
- `relation` 仅允许：`generalization`、`realization`
- `targetId` 必须引用同围栏中已声明的某个 `classes[].id`

### 5.4.2 `members[]`

- 表示“字段/成员变量”列表（复数字段名）
- 每项至少需要 `name`
- 可选：`visibility`、`type`、`static`、`accessor`、`associatedClassifierId`、`typeFromAssociation`、`notes`
- `associatedClassifierId` 若存在，必须指向已声明 `classes[].id`

### 5.4.3 `methods[]`

- 表示“方法”列表（复数字段名）
- 每项至少需要 `name`
- 可选：`visibility`、`virtual`、`methodKind`、`operatorSymbol`、`signature`、`type`、`typeFromAssociation`、`notes`
- `methodKind` 允许：`normal|constructor|destructor|functor|operator`
- 当 `methodKind === "operator"` 时，`operatorSymbol` 必须为非空字符串

### 5.4.4 `enums[]`

- 表示“枚举字面量”列表（复数字段名）
- 每项至少需要 `name`
- 可选：`enumGroup`、`type`、`notes`

### 5.4.5 `properties[]`

- 表示属性列表
- 每项至少需要 `name`
- 可选：
  - `backingFieldName`
  - `backingVisibility`
  - `associatedClassifierId`
  - `type`
  - `typeFromAssociation`
  - `static`
  - `hasGetter` / `hasSetter`
  - `getterVisibility` / `setterVisibility`
  - `notes`
- `getterVisibility` / `setterVisibility` 允许：`public|protected|private|package`
- `associatedClassifierId` 若存在，必须指向已声明 `classes[].id`

## 5.5 `associations[]`

- 每项必填：`id`、`kind`、`fromClassifierId`、`toClassifierId`
- `kind` 允许：`association|aggregation|composition|dependency`
- 两端点都必须指向同围栏中已声明 `classes[].id`
- 可选端点信息：
  - `fromEnd` / `toEnd`
  - 每个 End 可含：`role`、`multiplicity`、`navigable`

## 7. 升级建议

升级到本版本时，建议先对历史数据执行一次批量迁移：

- 将 `classes[].member` 重命名为 `classes[].members`
- 将 `classes[].method` 重命名为 `classes[].methods`
- 将 `classes[].enum` 重命名为 `classes[].enums`
- 清理 `classes[].members[{ kind: ... }]` 的 legacy 结构（按新结构拆分）

迁移后再通过 `parseMarkdownBlocks()` 做全量校验，确认无 schema 错误。

可选：使用仓库脚本自动迁移（支持 dry-run 与统计输出）：

- `npm run migrate:codespace-schema -- <path>`
- `npm run migrate:codespace-schema -- <path> --write`
